Intro to Sketchpad

In 1961, Ivan Sutherland built a computer program called Sketchpad. It was the first in many things: first graphical user interface (GUI), first time an oscilloscope was modified to be used as a computer monitor as we know it now, first gesture UI using a light pen, first computer drawing system. Still, to a casual observer it appears to be a CAD drawing tool, something like the great-great grandpa of AutoCAD. But programming languages pioneer Alan Kay points out some other important categories it is a first in, not commonly known. It happens to be really the first object oriented system, as well as first fully declarative programming system. That is, the user simply expresses the constraints (requirements) of the drawing (e.g. two line sections must be parallel) and the system will automatically and continually solves the system of constraints to ensure they are maintained at all times. Listen to Kay's video commentary here to get a better idea:

Sadly, from what I've seen 1961 Sketchpad happens to be not only the first truely declarative programming system (albeit a limited and not really extensible one), but also one of the last ones! Alan Borning's ThingLab of late 70s generalized it by packagaing it as a constraint-based "kit building kit," programmable within the Smalltalk environment. Other forms of declarative programming and constraint-based programming came about in later years. Language researchers have also tried various forms of mixing declarative, constraint-based programming with the common imperative style programming. In either case, I haven't seen any form of declarative programming system that can handle the scale of realistic software development of today. The grand majority of developers have never heard of declarative programming, let alone use it in their practice.


Constraint Solving in Sketchpad

An important innovation of Sketchpad was to employ an iterative numerical constraint solving approach called relaxation. Sketchpad system uses it along with a linear approximation method as a relatively light-weight scheme to solve for a given system of numeric constraints, including non-linear ones. The goal of the iterativive relaxation method is to gradually converge to a solution to a system of (maybe nonlinear) equations. Another aspec of relaxation technique is that an error value is used as an indicator for how far each constraint is from a solution. Once all constraints have a miniscule error value, the system is assumed to have converged to an acceptable global solution.

Because the system can, in many situations, smoothly and quickly converge to an acceptable solution, this form of constraint solving can be suitable for interactiveness and animation, as shown by Sketchpad and Borning's ThingLab. Sketchpad's relaxation method did not employ any domain specific knowledge in solving a system of constraints: only a general iterative linear approximation method.


Thirty Years Later...

In his paper published in the 90s "30 Years After Sketchpad: Relaxation of Geometric Constraints Revisited," C.W.A.M. van Overveld reworked Sketchpad's original relaxation solving model for a small set of geometric constraints, e.g., one to maintain a fixed distance from two points.

Rather than using a general linear-approximation based numeric method to solve constraints, Overveld proposed to let each type of constraint to define its own algorithm to solve itself, in isolation. Once the solutions from all individual constraints are collected, they are averaged together and applied to the variables. In a solution-converging situation, the system moves closer to a solution by this pass and therefore by iteratively repeating this process an acceptable global solution (miniscule error values) can be reached.

From one point of view Overveld's approach is inferior to Sketchpad's: the system has no real smarts in solving problems; the user has to define constraint types beforehand and also provide an algorithm that solves it.

What we noticed, however, is that from the standpoint of a programming language, Overveld's approach and way of organizing a program as a selection of constraints from a set of predefined constraint types is incredibly suitable for applying to general programming. It's nice that Sketchpad's general numeric relaxation technique doesn't require providing an algorithm to solve constraints, but it is also limited in being applable to most software engineering scenarios, e.g. non-numeric constraints. Another advatnage of Overveld's way is that it gives the user the power her own experties and knowledge to solve constraints whichever is most suitable.
